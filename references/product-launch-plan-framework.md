# 新品上架全案框架

本文件用于 Step 9 生成 `launch_plan`，适用于“参考竞品做新品上架方案”“把卖点转成买点”“规划五张主图和详情页执行稿”的任务。默认品类为灯具/照明，其他电商品类可沿用结构。

## 1. 输出定位

新品上架全案不是普通竞品报告，而是把竞品数据转成可执行上架动作：

`竞品数据 → 市场弱点 → 用户顾虑 → 我方超过竞品的打法 → 买点文案 → 主图/详情/SKU/验证`

必须回答：

- 如果要超过竞品，我方靠什么赢。
- 先做哪一个动作，为什么不是其他动作。
- 五张主图分别承担什么转化任务。
- 参数卖点如何改写成用户买点。
- 哪些表达存在合规风险，如何替换。

## 2. 固定 Markdown 结构

当用户要“新品上架全案/新品策划全案”时，最终 Markdown 必须按以下结构输出，不得只输出 `launch_plan` 摘要：

```markdown
# {类目/商品} · 天猫新品上架全案

> **Skill版本**：lighting-product-launch v{version}
> **数据来源**：{商品数}款商品，原始评价抽样 {review_count} 条 + 问答 {qa_count} 条
> **自家商品**：{店铺名}（ID：{商品ID}）

---

## 一、竞品市场概览
### 1.1 核心竞品数据
### 1.2 市场关键发现

## 二、卖点策划方案
### 2.1 竞品卖点现状分析
### 2.2 核心卖点策划（三层卖点体系）
#### 第一层：超级卖点（1个）
#### 第二层：重要卖点（3个）
#### 第三层：辅助卖点（3个）

## 三、SKU方案策划
### 3.1 现有SKU分析
### 3.2 新品SKU方案建议
### 3.3 SKU定价策略
### 3.4 SKU图片优化建议

## 四、卖点转化为买点：创意图文案
### 4.1 买点转化对照表（六步链路精简版）

### 4.3 五图买点创意图执行稿

## 六、合规注意事项
```

说明：

- 顶层章节必须保持“一、二、三、四、五、六”结构。
- 卖点策划必须放在 SKU方案之前；先从竞品弱点和用户顾虑里确定“我方靠什么赢”，再决定 SKU 怎么命名、定价和承接。
- 决策指导模块必须放在卖点转买点创意图文案之后，作为第五章；先让运营和美工看到完整的页面表达方案，再判断本轮优先做什么、暂缓什么、如何验证。
- `4.2 超级买点文案` 不得只输出两段代码块。最终 Markdown 必须包含：方案用途对比表、主图/搜索图的文案层级表、画面要求、禁用表达，以及效果示例图或效果示意说明。
- `4.3 五图买点创意图执行稿` 不得只输出五段代码块。必须先给“五张图分工表”，再按每张图输出：转化任务、用户买点、画面结构、文案层级、证据/合规提醒，以及效果示例图或示意图提示。
- `4.4 详情页核心买点文案` 不得只输出四段文案。必须按“详情页信任链”展示：每屏任务、用户疑问、内容模块、核心文案、不要放什么；优先用一张长屏示意图或长屏结构说明展示滑动节奏。
- 如果没有真实商品图，可以使用 SVG/PNG 示意图或“效果示例图生成提示”，但必须标注“示意图，不代表最终商品实拍”；正式出图应替换为真实安装/使用场景。
- 本地交付时，必须在 `output` 下创建一个独立交付目录，目录内只放最终 Markdown 报告和它引用的示例图片；图片链接使用 `fanlight_image1_main.svg` 这类同目录相对链接。Cloudflare R2/远程 URL 仅作为发布或分享场景的可选步骤，不作为默认输出。
- 如果没有足够数据支撑某个具体金额、服务时效或证书，只写“待补充/以页面服务条款为准”，不得编造。

## 3. 评价/问答证据规则

所有与用户反馈、评论、问答相关的洞察，必须从 `raw/*.json` 的原始 `feedback` 和 `question` 文本中提取，不能只引用 `analysis.json` 里的摘要字段。

必须遵守：

- 评价洞察必须能回到 `【竞品详细评价抽样】` 中的评价编号，例如“评价#68 提到换完灯后房子变化、舒服、不晃眼”。
- 问答洞察必须能回到 `【核心问答抽样】` 中的问答编号，例如“问答Q2 询问30多平买多大并反馈117cm亮度差点”。
- 可以统计原始文本中的关键词次数，但结论后必须至少附一个原始评价或问答样例。
- `【竞品全局印象标签】` 和 `【买家售前顾虑标签】` 只能作为辅助索引，不能单独作为最终洞察证据。
- 禁止写“用户反馈显示……”但不给评价编号、问答编号或原文来源。

## 5. 卖点转买点规则

不要从产品参数出发堆卖点，要从用户真实顾虑和购买场景出发：

- 参数转结果：`Ra≥97` → `开灯就有自然好气色`。
- 功能转利益：`离线声控` → `不用买天猫精灵，全家人都会用`。
- 服务转阻力解除：`免费安装` → `你只需要开门，其他全包`。
- 优惠转下单理由：`省¥466` → `这次一起换，比分批换更省`。

`buy_point_conversion` 每条建议包含：

- `user_inner_voice`：用户内心独白。
- `selling_point`：对应产品卖点。
- `scene_expression`：场景化买点文案。
- `landing_position`：主图/详情页/SKU/客服话术等落地位置。

## 6. 五张主图固定逻辑

1. 图1：颜值/第一购买理由，让用户停下来。
2. 图2：选购焦虑和核心功能证明。
3. 图3：独家体验或核心差异化。
4. 图4：安装、售后、质量等信任阻力。
5. 图5：套餐、价格、服务或下单理由。

每张图至少输出：

- `image_no`：图序。
- `buy_point`：用户买点。
- `visual_scene`：画面建议。
- `headline`：主标题。
- `supporting_copy`：辅助文案。
- `avoid`：禁用表达或画面。

## 7. 决策指导模块

`decision_guidance` 必须放在 SKU方案、卖点策划和卖点转买点创意图文案之后，用于让运营知道“为什么先做这件事”。这样运营先看清“卖什么、怎么卖、具体页面如何表达”，再判断本轮优先动作。

默认按以下顺序判断：

1. 搜索承接是否弱。
2. 首图点击理由是否足够。
3. SKU 是否降低选择成本。
4. 详情页是否建立信任链。
5. 评价/问答是否补齐下单顾虑。
6. 投放是否值得放大。

决策矩阵建议字段：

| 字段 | 说明 |
|---|---|
| `option` | 可选动作，如重做主图、改 SKU、降价、补详情页 |
| `conversion_impact` | 影响转化评分，1-5 |
| `competitor_scarcity` | 竞品稀缺评分，1-5 |
| `implementation_feasibility` | 自身可落地评分，1-5 |
| `compliance_risk` | 合规风险，low / medium / high |
| `score` | 综合判断 |
| `decision` | 做 / 暂缓 / 先补数据 |

禁止把多个方案平铺成“都可以”。必须给出唯一推荐路径，并说明本轮不做什么。

## 8. 合规规则

必须标注高风险表达和替代表达：

| 禁用表达 | 替代表达 |
|---|---|
| 全网最亮 / 最好 / 第一 | 用场景实拍和具体适用面积证明 |
| 狠大狠亮 / 超大超亮 | 尺寸 + 适用面积 |
| 保护视力 / 防近视 / 医学级 | Ra≥97 全光谱，光感接近自然光，附检测报告 |
| 100%好评 / 零差评 | 真实评价数量 |
| 专利技术 | 仅在有证书时使用 |

没有检测报告、证书或原始数据支撑时，只能写“建议表达方向”，不得写成确定事实。
